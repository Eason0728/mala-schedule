# 員工打卡功能 — 設計文件

日期：2026-07-05
狀態：待使用者確認

## 背景與目標

現有排班系統（`index.html`）純前端＋GitHub Gist 雲端同步，寫入需 GitHub token，因此無法讓每位員工直接寫入 Gist。本功能讓員工用**自己的手機**在**專屬連結**上打卡，記錄實際出勤時間，之後可用來：

1. 記錄出勤（誰、幾點上/下班）
2. 比對班表，標出遲到/早退/缺卡
3. 帶入薪資計算，取代（或對照）排定工時

**交付順序（使用者指定）**：
① 打卡系統（記錄，操作穩定可用）→ ② 薪資換算（用實際工時算薪水）→ ③ 班表連動（比對排定班表，抓遲到/早退）。

每個階段各自獨立可上線、可測試，不互相阻擋。本文件涵蓋三階段的整體架構與資料模型；階段二、三的細部規則（薪資規則、比對邏輯）在各自開始前另行釐清並可能更新本文件或另立 spec。

## 適用範圍

單一門市（不需跨店判斷）。分段班：一天可有多組上/下班（如中午一段、晚上一段），每次上/下班各打一次卡。

## 架構總覽

```
員工手機（clock.html，專屬連結）
   │  打卡：GPS 定位 + 裝置碼
   ▼
Google Apps Script（Web App，公開可寫入端點）
   │  驗證：門市範圍內、裝置已綁定（或走核准流程）
   ▼
Google 試算表
   │  原始打卡事件 + 員工名冊 + 裝置綁定狀態
   ▼
排班 App 後台（index.html 新增「出勤」分頁）
   讀回試算表資料顯示/管理，之後接薪資/班表比對
```

班表／員工／薪資資料仍走原本的 GitHub Gist 同步，兩套資料來源獨立，互不覆蓋。

### 為何用 Google Apps Script 當後端

- 現有雲端同步的寫入需要 GitHub token，不能發給每位員工。
- Apps Script Web App 可設「任何人皆可存取」，免登入即可 POST，且和使用者現有的 LINE 週報／績效評核／日報表系統同一套技術棧，維護成本低、免費。
- 資料落在 Google 試算表，使用者可直接開表核對，不必透過程式介面除錯。

## 認人與防作弊機制

| 機制 | 作用 | 擋不住的情況 |
|---|---|---|
| 專屬連結（每位員工一組含 key 的網址） | 免登入認出是哪位員工 | 連結被轉傳 |
| GPS 定位（單店座標 + 允許半徑） | 擋在家/外地打卡 | 人在店裡的代打 |
| 裝置綁定 | 擋連結轉傳給別人在別支手機打卡 | 同一支手機借給同事本人操作 |

**裝置綁定運作方式**：
- 員工第一次用手機開啟專屬連結時，前端產生一組裝置碼（存在該手機的 localStorage），連同打卡請求一起送出，後端記錄為該員工的「已綁定裝置」。
- 之後每次打卡都會帶裝置碼，後端比對是否為已綁定裝置；相符才允許寫入。
- 若從**新裝置**打卡（裝置碼不同，例如連結被轉傳、或員工換手機），該次打卡標記為「待核准」，不直接算數，需使用者在排班 App 後台手動核准後才會生效並改綁新裝置。
- 三個機制中，GPS 與裝置綁定分別擋不同的作弊路徑（在家打 vs 轉傳代打），兩者合併已涵蓋「合理防弊、不過度增加操作摩擦」的需求；**未設 PIN 密碼**，因為同一支手機本人操作時不需要多一道輸入。

## 資料模型（Google 試算表）

### 分頁：`roster`（員工名冊，由排班 App 同步產生）

| 欄位 | 說明 |
|---|---|
| emp_id | 對應 `mala_employees` 的員工 id |
| name | 員工姓名 |
| key | 專屬連結用的隨機金鑰（打卡連結＝`clock.html?k={key}`） |
| device_id | 目前綁定的裝置碼；空＝尚未綁定 |
| device_bound_at | 綁定時間 |
| active | 是否在職（同步時對應排班 App 的員工在職狀態） |

### 分頁：`events`（打卡事件，逐筆寫入，不覆蓋）

| 欄位 | 說明 |
|---|---|
| ts | 伺服器收到時間（ISO，避免手機時間被改） |
| emp_id | 員工 id |
| type | `in` 或 `out`（上班／下班） |
| lat / lng | 打卡當下座標 |
| distance_m | 與門市座標的距離（公尺） |
| within_range | 是否在允許半徑內（布林） |
| device_id | 打卡當下的裝置碼 |
| device_match | 是否與 roster 綁定的裝置相符 |
| status | `ok`（正常入帳）／`pending_device_approval`（新裝置待核准）／`rejected_out_of_range`（超出範圍，仍記錄但標記不算數） |

一天多組上下班＝同一 emp_id 當天出現多筆 `in`/`out` 交錯事件，事後依時間序配對成工作時段（配對邏輯留給後續讀取端處理，`events` 分頁本身只負責忠實記錄原始事件，不做配對運算，避免寫入時邏輯過重）。

### 門市設定（Apps Script 端設定值，非試算表資料）

- 門市座標：緯度 24.7838051、經度 121.015592（新竹市東區光復路一段435號）
- 允許半徑：50 公尺

## 排班 App 端整合（index.html）

- **員工分頁**：新增「同步名冊到打卡系統」按鈕，把在職員工的 id/姓名同步到 `roster` 分頁（含新增/更新 key，不覆蓋已綁定的 device_id）。
- **新增「出勤」分頁**：讀取 Apps Script 提供的查詢端點，顯示每位員工的打卡紀錄；待核准的新裝置在此顯示並可一鍵核准/拒絕。
- 沿用現有系統的唯讀/可編輯裝置模型觀念（`IS_LOCAL_DEV` 本機測試不影響打卡系統，因為打卡資料走 Apps Script 而非 Gist）。

## 階段規劃

### 階段一：打卡系統（本次實作目標）
- `clock.html` 打卡頁：員工開啟專屬連結，看到自己姓名＋「上班」/「下班」按鈕，按下後取得 GPS＋裝置碼送出。
- Apps Script Web App：接收打卡請求，驗證範圍與裝置，寫入 `events`，回傳結果（成功/超出範圍/待核准）。
- 名冊同步：排班 App 產生/管理 `roster`。
- 出勤分頁：排班 App 讀回顯示原始打卡紀錄＋裝置核准介面。
- **驗收標準**：員工能在自己手機上完成打卡，操作順暢無卡關；使用者能在排班 App 看到即時打卡紀錄並管理裝置核准。

### 階段二：薪資換算
- 待階段一操作穩定後才開始規劃細節，需與使用者確認：沒打卡怎麼算、時間是否抓整（如四捨五入至 15 分）、遲到/早退是否扣款、加班費是否改依實際工時計算。
- 目標：`calcSalary` 可選擇改用階段一累積的實際工時計算，與現行「依排定班表工時」算法可對照切換，不強制立即取代舊算法。

### 階段三：班表連動
- 待階段二穩定後開始。將打卡事件與當月排定班表比對，標示遲到、早退、缺卡、實際工時 vs 排定工時差異。
- 呈現位置與呈現方式（班表格內標記 or 獨立報表）於該階段開始前另行確認。

## 明確排除（本階段不做）

- 不做員工登入帳密系統（沿用「專屬連結認人」模式，與現有排班 App 的分享連結風格一致）。
- 不做多門市 GPS 判斷（目前僅一家店）。
- 不在階段一動到薪資計算或班表比對邏輯。
- 不設 PIN 密碼（已由 GPS ＋ 裝置綁定的組合取代）。

## 部署與維護原則（延續現有 skill 慣例）

- 開發階段不對正式 Google 試算表/Apps Script 做會覆蓋既有資料的操作；如需種子資料，先在測試用試算表/測試部署驗證。
- 排班 App（`index.html`）端的改動比照現有規則：先在本機（`localhost:8765`）測過，明確回報「本機可測」，經使用者同意才 push 部署。
- 打卡系統為新系統，Apps Script 部署網址、試算表 ID 等視為新建資產，**部署（建立正式 Apps Script Web App／試算表）前需先讓使用者確認**，不擅自建立正式雲端資源。
